# Feature Prompt Template

Use this whenever you are adding a new route or feature. The template drives
a six-stage process:

```
[Design]  →  [Phase 1]  →  [Phase 2]  →  [Phase 3]  →  [Phase 4]  →  [Audit]  →  [Tests]  →  [E2E]  →  [Docs]
```

Each stage is a separate file handed to the AI as a fresh session. The AI
re-reads the file system at the start of every stage. It never carries state
from a previous session.

---

## Package layout rule — one route, one folder

Every route lives in its own package folder. A folder never contains more than
one HTTP endpoint. This prevents handler and service files from growing
unboundedly as new `/me` routes are added.

```
internal/domain/auth/
├── profile/            ← PATCH /me/profile
├── identities/         ← GET /me/identities
├── set-password/       ← POST /set-password
├── username/           ← PATCH /me/username  +  GET /username/available
│                          (two endpoints, one package — availability check is a
│                           stateless helper for the same resource; stays together)
├── email/
│   ├── request/        ← POST /me/email/request-change
│   ├── verify/         ← POST /me/email/verify-current
│   └── confirm/        ← POST /me/email/confirm-change
├── delete/             ← DELETE /me
├── bootstrap/          ← POST /owner/bootstrap
├── magiclink/          ← GET /magic-link/verify
└── oauth/
    ├── google/         ← GET + GET/callback + POST/link + DELETE/unlink
    └── telegram/       ← POST/callback + POST/link + DELETE/unlink
```

**When one folder is allowed to hold more than one endpoint:**
- The endpoints share the exact same resource (e.g. OAuth provider: initiate +
  callback + link + unlink all operate on the same `user_identities` row).
- The availability check for a resource lives in the same folder as the mutation
  for that resource (e.g. `GET /username/available` + `PATCH /me/username`).

Any time you are unsure, give it its own folder. Splitting later is cheaper
than maintaining a bloated file now.

## Stage overview

| Stage | Name | Produces | Gate |
|---|---|---|---|
| **0 — Design** | Design & Decisions | Signed-off spec: decisions, HTTP contract, all test cases | Human review + approval |
| **1 — Foundations** | SQL + Types | Schema, audit event, models, request/response types | `go build ./internal/db/...` |
| **2 — Data layer** | Store | Store method(s), FakeStorer, QuerierProxy | `go build ./...` |
| **3 — Logic layer** | Service | Service method(s), FakeServicer, service unit tests | `go test` (unit, all cases pass) |
| **4 — HTTP layer** | Handler + Routes | Handler, routes, handler unit tests, store integration tests | `go test` (all) + §3.13 checklist |
| **5 — Audit** | Pre-test verification | Full cross-check: all paths covered, no gaps, no TODOs | AI sign-off doc |
| **6 — Unit tests** | Run all tests | Clean pass: unit + integration | `go test ./internal/domain/...` |
| **7 — E2E** | E2E collection | Newman collection written by AI; run and approved by human | `make e2e-{feature}` passes |
| **8 — Docs** | mint/ docs | New or updated `.mdx` files | Human review |

**Stage 0 is mandatory.** Do not write any code until Stage 0 is complete and
every open question has an answer. Undecided design questions at implementation
time produce inconsistent guards, undocumented error codes, and missing test cases.

## How to use this template

1. Copy the requirement text from `docs/map/INCOMING.md` for the route you are building.
2. Identify the affected files using the file map below.
3. Create `docs/prompts/{feature}/0-design.md` using the Stage 0 template.
4. Hand `0-design.md` to the AI. Review and approve the output.
5. Create `docs/prompts/{feature}/1-foundations.md`, etc. using the per-stage templates.
6. Hand one file to the AI per session. Do not combine stages.
7. After Stage 4 passes, run Stage 5 (audit) as an AI session — it reads all
   produced files and confirms no paths or checks were missed.
8. Run Stage 6 (unit tests) yourself — no AI session needed.
9. Open a new AI session with `7-e2e.md` — the AI writes the Newman collection
   and Makefile target. You run it and approve.
10. Create `docs/prompts/{feature}/8-docs.md` and hand it to the AI for the mint/ update.

## Audit checklist

Work through every item. For each, cite the exact file and line (or function
name) that satisfies it. If the item cannot be satisfied, mark it **FAIL** and
describe what is missing.

### A — HTTP contract

- [ ] Method and path in `routes.go` match Stage 0 §2 exactly
- [ ] Auth middleware present (or intentionally absent with a documented reason)
- [ ] Request body fields match Stage 0 §2 — no extra, no missing
- [ ] Success response shape matches Stage 0 §2 — correct status code and fields
- [ ] Every error code from the Stage 0 §2 error table appears in the handler error switch
- [ ] No error code in the switch is absent from Stage 0 §2 (no undocumented codes)

### B — Guard ordering

- [ ] Service method implements every guard from Stage 0 §5 in the documented order
- [ ] No guard has been silently dropped
- [ ] No guard has been added that was not in Stage 0 §5 (undocumented checks are bugs)

### C — Security

- [ ] Timing-invariant paths (Stage 0 §5) use a dummy hash or equivalent; confirm
      the dummy call is never skipped by early return
- [ ] `context.WithoutCancel` is used for every write marked in Stage 0 §5;
      confirm no critical write can be skipped by client disconnect
- [ ] OAuth/HMAC or token validation (if applicable) is not bypassable
- [ ] No raw sensitive value is logged or returned to the caller

### D — Audit

- [ ] Every audit event from Stage 0 §4 is emitted at the correct point in the
      service method
- [ ] No audit event is emitted before the operation it records has succeeded
- [ ] `context.WithoutCancel` wraps the audit write (so a disconnect cannot silently
      skip the audit row)

### E — Rate limiting

- [ ] Every rate limiter from Stage 0 §6 is registered in `routes.go`
- [ ] Every KV prefix from Stage 0 §6 is present in the limiter constructor
- [ ] No prefix collides with any prefix in `E2E_CHECKLIST.md`

## Guidance: writing the test case inventory (Stage 0 §7)

The test case inventory is the most important output of Stage 0. It is the
contract between design and implementation. If a path is not in the inventory,
it will not be tested.

**How to derive cases from the guard ordering:**

For a service with this guard sequence:
```
1. Fetch token — ErrTokenNotFound if absent
2. Check expiry — ErrTokenExpired if past
3. Check attempts — ErrTooManyAttempts if budget exhausted
4. Check code hash — ErrInvalidCode if wrong
5. Mutate
6. Audit write
```

The service test cases are:

| Case | How it's reached | Assert |
|---|---|---|
| Happy path | All guards pass | Returns success value, nil |
| ErrTokenNotFound | Store returns not-found | errors.Is(err, ErrTokenNotFound) |
| ErrTokenExpired | Token.ExpiresAt in past | errors.Is(err, ErrTokenExpired) |
| ErrTooManyAttempts | Token.Attempts == Token.MaxAttempts | errors.Is(err, ErrTooManyAttempts), increment NOT called |
| ErrInvalidCode (budget remains) | Wrong code, Attempts < Max | errors.Is(err, ErrInvalidCode), increment called |
| Timing invariant: dummy hash on not-found | Store returns not-found | Dummy hash called exactly once |
| context.WithoutCancel on increment | Wrong code path | Captured ctx has nil Done() channel |
| Increment failure swallowed | IncrementFn returns error | Still returns ErrInvalidCode |
| Correct audit event sent to increment | Wrong code path | capturedIn.AttemptEvent == expected |
| Store error wraps | Store returns raw error | err.Error() contains "service.Method:" |

For each handler error case:
- Map each sentinel → HTTP status (from the error switch)
- Add one test per sentinel
- Add validation failure cases (empty field, wrong format)
- Add the nil-infrastructure case if handler has an infrastructure dependency

For each integration test:
- DB state: not found → correct error
- DB state: found, valid → returns populated result
- DB state: found but consumed/expired → correct error
- QuerierProxy Fail flag → ErrProxy returned

**Rules for test cases:**
- Every sentinel error that can be returned by the service must have an H-layer case.
- Every sentinel error that can be returned by the store must have an I-layer case.
- Every `context.WithoutCancel` usage must have an S-layer case asserting the detached context.
- Every timing-invariant dummy call must have an S-layer case counting invocations.
- Every "single-use" resource (KV token, DB token) must have an H or I case asserting deletion.
- Every rate-limiter added must appear in the Stage 0 §6 table.
