## Key context (extracted from design — do not re-read files if you already have these)

### HTTP contract summary

**DELETE /api/v1/profile/me** — JWT required

| Path | Trigger | Success |
|---|---|---|
| A — password user | `"password"` field present | `200 {"message":"account scheduled for deletion","scheduled_deletion_at":"<RFC3339>"}` |
| B-1 — email-OTP step 1 | empty body, user has email, no password | `202 {"message":"verification code sent to your email"}` |
| B-2 — email-OTP step 2 | `"code"` field present | `200 {"message":"account scheduled for deletion","scheduled_deletion_at":"<RFC3339>"}` |
| C-1 — Telegram step 1 | empty body, user has no email, has Telegram identity | `202 {"message":"authenticate via Telegram to confirm deletion","auth_method":"telegram"}` |
| C-2 — Telegram step 2 | `"telegram_auth"` field present | `200 {"message":"account scheduled for deletion","scheduled_deletion_at":"<RFC3339>"}` |

Error codes for DELETE /me: `already_pending_deletion` (409), `invalid_credentials` (401),
`invalid_telegram_auth` (401), `telegram_identity_mismatch` (401), `validation_error` (400/422),
`token_not_found` (422), `token_already_used` (422), `invalid_code` (422), `too_many_attempts` (429),
`too_many_requests` (429), `unauthorized` (401), `internal_error` (500).

**POST /api/v1/profile/me/cancel-deletion** — JWT required, no body

Success: `200 {"message":"account deletion cancelled"}`

Error codes: `not_pending_deletion` (409), `too_many_requests` (429), `unauthorized` (401),
`internal_error` (500).

### Rate limiters (both user-keyed, INSIDE JWT middleware group)

| Endpoint | Prefix | Limit |
|---|---|---|
| DELETE /me | `del:usr:` | 3 req / 1 hr per user |
| POST /me/cancel-deletion | `delc:usr:` | 5 req / 10 min per user |

**Critical invocation decision:** Both limiters are **inside** the JWT middleware group.
Running rate-limiting in a separate Newman invocation loses the access token stored in
collection variables, causing 401 instead of 429. All folders — including both
rate-limiting folders — must run in **one single Newman invocation**. Use dedicated
RL-user Gmail aliases with fresh buckets (same +alias pattern as set-password, email-change).

### Sentinel errors and their HTTP status codes (for error switch assertions)

| Sentinel | Status | Code |
|---|---|---|
| `ErrAlreadyPendingDeletion` | 409 | `already_pending_deletion` |
| `ErrNotPendingDeletion` | 409 | `not_pending_deletion` |
| `ErrInvalidCredentials` | 401 | `invalid_credentials` |
| `ErrInvalidTelegramAuth` | 401 | `invalid_telegram_auth` |
| `ErrTelegramIdentityMismatch` | 401 | `telegram_identity_mismatch` |
| `ErrTokenNotFound` | 422 | `token_not_found` |
| `ErrTokenAlreadyUsed` | 422 | `token_already_used` |
| `ErrInvalidCode` | 422 | `invalid_code` |
| `ErrTooManyAttempts` | 429 | `too_many_attempts` |

### Session non-revocation intent (D-02)

After a successful soft-delete (any path returning 200), the JWT remains valid.
`GET /api/v1/profile/me` must still return `200` (not 401) and the response must
include `scheduled_deletion_at`. Assert this via a follow-up GET after every
successful delete path.

### Login for pending-deletion accounts (D-04 / T-34)

After soft-delete, `POST /auth/login` with the same credentials must return `200`
and the response body must include `scheduled_deletion_at`. Assert this in the
`happy-path-password` folder.

### Cancel reverses the deletion (T-27)

After `POST /me/cancel-deletion` succeeds, `GET /me` must return `200` with
`scheduled_deletion_at` absent (`omitempty`). Assert this in `happy-path-cancel`.

##### Folder 1: `setup`

Creates the two user accounts needed for happy-path tests.

**Steps:**
1. Read `gmail_test_email` from environment; set `del_email`, `del_password = "P@ssw0rd1!"`.
2. `POST /auth/register` with `del_email` → 201. Assert status 201.
3. Gmail OTP fetch (prerequest busy-loop wait 8 s, query `subject:"Verify" to:del_email newer_than:1h`; extract 6-digit code; store as `del_verify_otp`).
4. `POST /auth/verify-email {"code": del_verify_otp}` → 200.
5. `POST /auth/login {"identifier": del_email, "password": del_password}` → 200. Store `del_access_token`.
6. Set `del_b_email = gmail_test_email + "+delb@..."` (derive alias from `gmail_test_email` using the same string-split pattern used by other collections), `del_b_password = "P@ssw0rd1!"`.
7. `POST /auth/register` with `del_b_email` → 201.
8. Gmail OTP fetch for `+delb` (filter `to:del_b_email`); store as `del_b_verify_otp`.
9. `POST /auth/verify-email {"code": del_b_verify_otp}` → 200.
10. `POST /auth/login {"identifier": del_b_email, "password": del_b_password}` → 200. Store `del_b_access_token`.

##### Folder 10: `rate-limiting-delc`

Confirms `delc:usr:` (5 req / 10 min per user) fires correctly.

**Dedicated alias user:** `gmail_test_email+cncrl@...`. Register + verify + login first.

**Prerequest:** pin `_xff = "127.0.0.1"` for all requests in this folder.

**Strategy:** Soft-delete then cancel, repeating 5 times with re-deletes in between.
The `del:usr:` limit (3/hr) will be exhausted after the 3rd re-delete. Accept that
re-delete requests may return 429 after that — only assert on the 6th cancel being 429.

| Request | Key assertions |
|---|---|
| `POST /register` `+cncrl` → 201 | setup |
| Gmail OTP + verify + login `+cncrl`; store `cncrl_access_token` | setup |
| `DELETE /me {"password": cncrl_password}` → accept 200 or 429 (del bucket) | prime for cancel |
| `POST /me/cancel-deletion` (warmup 1) → 200 | 1/5 used; accept 409 if delete above 429'd |
| `DELETE /me` → accept 200 or 429 | re-prime |
| `POST /me/cancel-deletion` (warmup 2) → accept 200 or 409 | 2/5 |
| `DELETE /me` → accept 200 or 429 | |
| `POST /me/cancel-deletion` (warmup 3) → accept 200 or 409 | 3/5 |
| `DELETE /me` → accept 200 or 429 | |
| `POST /me/cancel-deletion` (warmup 4) → accept 200 or 409 | 4/5 |
| `DELETE /me` → accept 200 or 429 | |
| `POST /me/cancel-deletion` (warmup 5) → accept 200 or 409 | 5/5 |
| `POST /me/cancel-deletion` (6th request) → 429 | **Hard assert:** status 429; `code == "too_many_requests"`; `Retry-After` header present |

*The intermediate warmups use `accept 200 or 409` because the account may or may not be
in pending-deletion state depending on whether the preceding DELETE succeeded. Only the
final 429 is the meaningful assertion — it proves the `delc:usr:` bucket is wired.*

#### Collection-level prerequest (mandatory — copy exactly from existing collections)

```javascript
// Increment the request sequence counter and derive a unique X-Forwarded-For IP
// for every request so non-RL folders never exhaust any rate-limit bucket.
// The rate-limiting folders override _xff to 127.0.0.1 in their own prerequest.
// Requires TRUSTED_PROXIES=127.0.0.1/32 in .env.
const n = (pm.collectionVariables.get('_req_seq') | 0) + 1;
pm.collectionVariables.set('_req_seq', n);
const ip = '10.' + ((n>>16)&0xff) + '.' + ((n>>8)&0xff) + '.' + (n&0xff);
pm.collectionVariables.set('_xff', ip);
```

Rate-limiting folder prerequest override (add to each RL folder individually):
```javascript
// Pin XFF to 127.0.0.1 for rate-limit testing.
// The del:usr: and delc:usr: limiters are user-keyed so this has no effect on
// bucketing, but it maintains the convention that RL folders always pin their IP.
pm.collectionVariables.set('_xff', '127.0.0.1');
```
