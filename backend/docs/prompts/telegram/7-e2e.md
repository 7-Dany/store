## Deliverable 1 — `e2e/oauth/telegram.json`

### Variable prefix

All collection variables use the `tg_` prefix (e.g. `tg_access_token`, `tg_callback_id`, `tg_auth_date`).

### HMAC computation (critical — read carefully)

Unlike Google OAuth, Telegram uses a widget-based HMAC flow: **you can compute
a valid HMAC in the test script** because the bot token is available in the
environment as `telegram_bot_token`. The Postman/Newman sandbox exposes
`CryptoJS` globally. Use it as follows in any prerequest script that needs to
build a valid callback payload:

```javascript
// Telegram HMAC algorithm (Stage 0 §2 / validators.go):
//   secret_key        = SHA256(botToken)           — CryptoJS.SHA256(botToken)
//   data_check_string = sorted "key=value" pairs   — exclude "hash" field
//   hash              = hex(HMAC_SHA256(secret_key, data_check_string))
const botToken = pm.environment.get('telegram_bot_token');
const secretKey = CryptoJS.SHA256(botToken);           // WordArray
const tgId      = pm.collectionVariables.get('tg_callback_id');
const authDate  = Math.floor(Date.now() / 1000);

// Build sorted pairs — add optional fields only when non-empty.
const pairs = [`auth_date=${authDate}`, `id=${tgId}`].sort();
const dataCheckString = pairs.join('\n');

const hash = CryptoJS.HmacSHA256(dataCheckString, secretKey)
                     .toString(CryptoJS.enc.Hex);

pm.collectionVariables.set('tg_auth_date', authDate);
pm.collectionVariables.set('tg_hash', hash);
```

Store `tg_callback_id` as a fixed test integer (e.g. `99887766`) in the
collection-level variable defaults. Use a different integer (e.g. `99887700`)
for the link-happy-path test so that the two flows don't collide.

### Folder structure and invocation strategy

All link and unlink endpoints sit **inside** the JWT middleware group, so their
JWT access token is lost between separate Newman invocations. The callback
endpoint is public (IP-rate-limited only). Therefore:

**Invocation 1 — single Newman run (setup through rate-limiting-lnk/unlk):**

| Folder | Purpose |
|---|---|
| `setup` | Register `gmail_test_email`, fetch OTP from Gmail, verify email, login → store `tg_access_token` |
| `callback-happy-path` | New user callback → 201 + access_token body + HttpOnly refresh_token cookie; same ID again → 200 |
| `callback-failures` | Invalid HMAC → 401 `invalid_signature`; auth_date 90 000 s old → 401 `auth_date_expired`; auth_date 120 s in future → 401 `auth_date_expired`; id=0 → 422 `validation_error`; malformed JSON → 400/422; account locked → 423 `account_locked`; account inactive → 403 `account_inactive` |
| `link-happy-path` | POST /oauth/telegram/link with `tg_access_token` and a fresh Telegram ID → 204; verify by checking GET /profile/me/identities includes provider=telegram |
| `link-failures` | Already linked → 409 `provider_already_linked`; invalid HMAC → 401 `invalid_signature`; auth_date expired → 401 `auth_date_expired`; malformed JSON → 400/422 |
| `unlink-happy-path` | DELETE /oauth/telegram/unlink → 204; setup user has a password so last-auth-method guard does not fire |
| `unlink-failures` | Provider not linked (just unlinked) → 404 `provider_not_linked`; last-auth-method: create a fresh Telegram-only user via callback, then attempt unlink → 409 `last_auth_method` |
| `auth-failures` | POST /link with no Authorization header → 401; DELETE /unlink with no Authorization header → 401 |
| `validation` | POST /callback body > 1 MiB → 413; POST /link body > 1 MiB → 413 |
| `rate-limiting-lnk` | Warmup: 5 POST /link requests with valid HMAC but body that hits a guard (e.g. already-linked) — they consume a rate-limit slot even though they return 409. 6th request → 429 `too_many_requests` + `Retry-After: 900`. Folder prerequest sets `_xff` to `127.0.0.1`. |
| `rate-limiting-unlk` | Warmup: 5 DELETE /unlink requests — provider_not_linked (404) still consumes a slot. 6th → 429 + `Retry-After: 900`. Folder prerequest sets `_xff` to `127.0.0.1`. |

**Invocation 2 — after Redis flush (fresh `tgcb:ip:` bucket):**

| Folder | Purpose |
|---|---|
| `rate-limiting-cb` | Warmup: 10 POST /callback requests with invalid HMAC (rejected at handler guard — still consumes IP slot). 11th → 429 `too_many_requests` + `Retry-After: 60`. Folder prerequest sets `_xff` to `127.0.0.1`. |

### Account-locked and account-inactive test users

To test the 423 and 403 paths in `callback-failures`, you need users whose
accounts are locked or inactive. The simplest approach that avoids needing
admin endpoints:

- **Locked user:** Insert directly via SQL in the `_e2e-db-clean` step — but
  that's complex. Instead, document in the collection description that these two
  cases are **not covered here** because creating locked/inactive users requires
  admin tooling not yet present in the E2E suite. Mark them as `// NOT covered here`.
  They are covered by `handler_test.go` T-07 (locked) and T-08 (inactive).

### Collection-level conventions (mandatory — matches existing collections)

- Global `prerequest` script: auto-increment `_req_seq`, derive unique `10.x.x.x` XFF IP, set `_xff`.
- Every rate-limiting folder's own prerequest script overrides `_xff` to `127.0.0.1`.
- Every test script includes `// NOT covered here` comments for intentionally
  omitted cases.
- Every assertion comment explains **what** is being asserted and **why** it
  confirms the wiring (not just "status is 204").

### 2. Add `e2e-oauth-telegram` target

Follow the exact style of `e2e-oauth-google`. Key differences:

- The callback rate limiter is public (separate invocation like `e2e-oauth-google`'s init and cb limiters).
- The link and unlink rate limiters are user-scoped (single invocation — JWT must survive).
- Therefore: one combined invocation for folders `setup` through `rate-limiting-unlk`, then a second invocation (after Redis flush) for `rate-limiting-cb`.

```makefile
e2e-oauth-telegram: _e2e-check-env ## Run Telegram OAuth E2E (POST /callback + POST /link + DELETE /unlink)
```

Add both the Windows (PowerShell) and non-Windows (bash) branches.

### 3. Add `e2e-oauth-telegram` to `e2e-auth`

In the `e2e-auth` target, add `@$(MAKE) e2e-oauth-telegram` immediately after
`@$(MAKE) e2e-oauth-google` (both Windows and non-Windows branches).

### 4. Update `_e2e-db-clean`

Telegram users are registered with `email = ''` (empty string, not NULL — see
`store.go:OAuthRegisterTx`). They are identified by their `user_identities` row
with `provider = 'telegram'` and a known test `provider_uid` value.

Add the following clause to the `psql` DELETE command in `_e2e-db-clean` to
remove Telegram E2E test users:

```sql
OR id IN (
    SELECT user_id FROM user_identities
    WHERE provider = 'telegram'
    AND provider_uid IN ('99887766', '99887700')
)
```
