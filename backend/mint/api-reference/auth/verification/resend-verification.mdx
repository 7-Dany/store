---
title: "Resend Verification"
authMethod: "none"
description: "Issues a new email verification OTP, invalidating any previously unused tokens for the account."
api: "POST http://localhost:8080/api/v1/auth/resend-verification"
---

<ParamField body="email" type="string" required>
  The email address associated with the unverified account. Normalised to lowercase before lookup.
</ParamField>

## Behaviour

This endpoint always returns `202 Accepted` regardless of whether the email is registered, already verified, locked, or within a cooldown window. This is intentional — the uniform response prevents account enumeration.

**OTP lifetime** — Each code is valid for **15 minutes** from the moment it was issued. After 15 minutes the code expires and the user must request a new one.

**Resend cooldown** — A new code can be requested at most once every **2 minutes**, measured from when the last token was issued. Clicking verify — whether right or wrong — has no effect on the cooldown; only issuing a token resets it. The server always responds `202` regardless of whether the resend was accepted or silently suppressed.

All prior unused tokens for the account are invalidated before a new one is created.

<AccordionGroup>
  <Accordion title="Frontend implementation">
    Disable the resend button the moment a code is issued (registration or resend) and re-enable it after **2 minutes**. The button can then be used every 2 minutes for the full 15-minute token lifetime:

    ```
    T+0:00  Code issued — resend disabled
    T+2:00  Resend re-enables — first resend available
    T+4:00  Second resend available
    T+6:00  Third resend available
    ...     Every 2 minutes until T+15:00
    T+15:00 Token expired — resend issues a fresh 15-minute token
    ```

    Because the server gives no signal distinguishing a suppressed resend from a successful one, the client must drive the 2-minute timer entirely from the timestamp of the last issued code.
  </Accordion>
</AccordionGroup>

## Rate Limiting

**3 requests per 10 minutes per IP address**.

Clients that exceed this limit will receive a `429 Too Many Requests` response with a `Retry-After` header indicating when they may retry.

<RequestExample>
```json Example
{
  "email": "alice@example.com"
}
```
</RequestExample>

<ResponseExample>

```json 429

```

## Responses

<AccordionGroup>
  <Accordion title="202 — Accepted">
    The request was accepted. If the email belongs to an unverified account that is not locked and not within the resend cooldown window, a new code has been sent. In all other cases (unknown email, already verified, locked, within cooldown) the response is identical — no hint is given about the account's state.

    ```json
    {
      "message": "if that email is registered and unverified, a new code has been sent"
    }
    ```
  </Accordion>

  <Accordion title="422 — Validation Error">
    The request body failed structural validation before any account lookup was attempted. This is the only response that confirms something went wrong on the client side.

    ```json
    {
      "code": "validation_error",
      "message": "email is required"
    }
    ```
  </Accordion>

  <Accordion title="429 — Rate Limited">
    The IP address has exceeded the rate limit of 3 requests per 10 minutes. Check the `Retry-After` response header for how many seconds to wait before retrying.

    ```json
    {
      "code": "too_many_requests",
      "message": "too many requests — please slow down"
    }
    ```
  </Accordion>
</AccordionGroup>
