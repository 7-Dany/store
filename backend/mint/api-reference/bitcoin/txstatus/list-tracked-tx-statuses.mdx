---
title: "List Tracked Tx"
description: "List durable Bitcoin transaction-tracking rows for the authenticated user."
api: "GET http://localhost:8080/api/v1/bitcoin/tx"
---

<ParamField query="address" type="string">
  Optional Bitcoin address filter. Must be valid for the configured network.
</ParamField>

<ParamField query="txid" type="string">
  Optional 64-character hex txid filter.
</ParamField>

<ParamField query="tracking_mode" type="string">
  Optional tracking-mode filter. Allowed values: `txid`, `watch`.
</ParamField>

<ParamField query="limit" type="integer">
  Optional page size. Must be between `1` and `100`. Default: `100`.
</ParamField>

<ParamField query="cursor" type="string">
  Optional opaque keyset cursor returned by a previous page in `next_cursor`.
</ParamField>

## Behaviour

Returns saved tracked-transaction records owned by the authenticated user.

- `tracking_mode = "txid"` rows were created explicitly through [`POST /bitcoin/tx`](/api-reference/bitcoin/txstatus/create-tracked-tx-status).
- `tracking_mode = "watch"` rows were created automatically when one of the
  user's watched addresses matched a transaction.

Rows are returned with the most recently confirmed transactions first. For rows
that are not yet confirmed, the first-seen time is used instead.

Pagination is keyset-based:

- omit `cursor` for the first page
- pass the previous response's `next_cursor` to continue
- stop when `next_cursor` is omitted

The `cursor` value is opaque. Clients must store and replay it exactly as returned.
