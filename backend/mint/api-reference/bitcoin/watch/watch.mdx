---
title: "Watch Addresses"
description: "Register Bitcoin addresses to receive real-time transaction events on the SSE stream."
api: "POST /bitcoin/watch"
---

<ParamField body="network" type="string" required>
  The Bitcoin network the addresses belong to. Must match the server's configured
  network (e.g. `"testnet4"` or `"mainnet"`). Requests where this value differs
  from the server's `BTC_NETWORK` are rejected with `400 network_mismatch`.
</ParamField>

<ParamField body="addresses" type="string[]" required>
  Array of Bitcoin addresses to register for monitoring. Between 1 and 20
  addresses per request.

  <Expandable title="requirements">
    - Minimum 1 address, maximum 20 addresses per request.
    - Each address must be valid for the specified network.
    - Bech32 addresses (`bc1*` / `tb1*`) are normalised to lowercase before storage.
    - Base58Check addresses (P2PKH / P2SH) are stored in their original mixed-case form.
    - Submitting an address that is already registered is idempotent — it is
      returned in `watching` with no error and no side effects.
    - Each user may have at most **100 addresses** registered in total across all
      requests. Attempting to exceed this limit returns `400 watch_limit_exceeded`
      with `reason: count_cap`.
  </Expandable>
</ParamField>

## Behaviour

**Purpose** — This endpoint is the registration step for the SSE events stream
([`GET /bitcoin/events`](/api-reference/bitcoin/events/events)). Addresses registered
here are matched against incoming Bitcoin transactions; matching transactions are
delivered to the user's open SSE connection as `mempool_tx`, `confirmed_tx`, and
`mempool_tx_replaced` events. Calling `POST /watch` without an open SSE connection
is valid but the events will have no consumer until one is opened.

**Storage and expiry** — Registered addresses are held ephemerally, not persisted
to the database. A user's watch set expires after **30 minutes of inactivity**.
When the watch set expires while an SSE connection is open, the server emits a
`stream_requires_reregistration` event on the stream. The client must then re-call
this endpoint to resume receiving events. Calling this endpoint — even with
already-registered addresses — resets the inactivity timer.

**Idempotency** — Re-submitting an address that is already registered succeeds silently
and refreshes the inactivity timer. The address is included in the `watching` array
with no error. No audit event is written and no metrics are incremented.

**`watching` is not the full watch list** — The response echoes only the addresses
from the current request. It does not represent the user's complete set of registered
addresses. Clients must track their own full list.

**Address normalisation** — Bech32 addresses are lowercased before storage and
before being returned in `watching`. Base58Check addresses (P2PKH / P2SH) are
returned in their original casing.

**Per-user address capacity** — Each user may have at most **100 addresses** registered
in total across all requests. This cap is enforced atomically. Once reached, further
registrations return `400 watch_limit_exceeded` with `reason: count_cap` until addresses
are removed or an admin resets the user's registration window.

**`watch_limit_exceeded`** — This error has two distinct causes identified by the
`reason` field:

| `reason` | Meaning |
|---|---|
| `count_cap` | The user has reached their **100-address** watch limit. |
| `registration_window_expired` | The 7-day registration window has lapsed; requires an admin reset. |

## Rate Limiting

**10 requests per 1 minute per IP address**.

The rate limiter runs **before** JWT authentication. Requests that exceed the
limit are rejected with `429` before token validation occurs.

Clients that exceed this limit will receive a `429 Too Many Requests` response
with a `Retry-After` header indicating when they may retry.

## Responses

<AccordionGroup>
  <Accordion title="200 — OK">
    The addresses were registered successfully (or were already registered).
    `watching` contains the normalised form of each address from the request.

    ```json
    {
      "watching": [
        "tb1qw508d6qejxtdg4y5r3zarvary0c5xw7kxpjzsx",
        "mipcBbFg9gMiCh81Kj8tqqdgoZub1ZJRfn"
      ]
    }
    ```

    | Field | Type | Notes |
    |---|---|---|
    | `watching` | string[] | Normalised addresses from the current request that are now active |
  </Accordion>

  <Accordion title="400 — Bad Request">
    One of several validation failures. Inspect `code` to determine the cause.

    **`network_mismatch`** — The `network` field does not match the server's configured network.
    ```json
    {
      "code": "network_mismatch",
      "message": "request network does not match server network"
    }
    ```

    **`too_few_addresses`** — The `addresses` array is empty.
    ```json
    {
      "code": "too_few_addresses",
      "message": "at least one address is required"
    }
    ```

    **`too_many_addresses`** — More than 20 addresses were submitted.
    ```json
    {
      "code": "too_many_addresses",
      "message": "at most 20 addresses may be submitted per request"
    }
    ```

    **`invalid_address`** — One or more addresses failed format validation for the specified network.
    ```json
    {
      "code": "invalid_address",
      "message": "one or more addresses failed validation"
    }
    ```

    **`watch_limit_exceeded`** — The user's watch limit has been reached. Check `reason` for details.
    ```json
    {
      "code": "watch_limit_exceeded",
      "message": "watch address limit exceeded",
      "reason": "count_cap"
    }
    ```
    ```json
    {
      "code": "watch_limit_exceeded",
      "message": "watch registration window expired",
      "reason": "registration_window_expired"
    }
    ```

    **`bad_request`** — The request body could not be parsed as valid JSON.
    ```json
    {
      "code": "bad_request",
      "message": "invalid request body"
    }
    ```
  </Accordion>

  <Accordion title="401 — Unauthorized">
    No valid JWT was provided or the user ID could not be extracted from the token.

    ```json
    {
      "code": "unauthorized",
      "message": "authentication required"
    }
    ```
  </Accordion>

  <Accordion title="413 — Request Entity Too Large">
    The request body exceeds the 1 MiB limit.
  </Accordion>

  <Accordion title="429 — Rate Limited">
    The IP address has exceeded 10 requests per 1 minute. Check the `Retry-After`
    response header for the number of seconds to wait before retrying.

    Note: this limit is enforced **before** JWT authentication — the request never
    reaches token validation when the IP bucket is exhausted.

    ```json
    {
      "code": "rate_limit_exceeded",
      "message": "too many requests"
    }
    ```
  </Accordion>

  <Accordion title="500 — Internal Server Error">
    An unexpected error occurred.

    ```json
    {
      "code": "internal_error",
      "message": "internal server error"
    }
    ```
  </Accordion>

  <Accordion title="503 — Service Unavailable">
    The Redis back-end is temporarily unavailable. Retry after the number of
    seconds indicated by the `Retry-After` response header (fixed at 5 seconds).

    ```json
    {
      "code": "service_unavailable",
      "message": "service temporarily unavailable"
    }
    ```
  </Accordion>
</AccordionGroup>
