<Note>
  Not sure which flow applies to your user? Call [`GET /profile/me/deletion`](/api-reference/profile/deletion/deletion-method) first.
  For a full UI walkthrough see the [Account Deletion Flow guide](/guides/profile/delete-account-flow).
</Note>

<ParamField body="password" type="string">
  The account's current password. Send this to confirm deletion immediately
  (Path A — password accounts only).
</ParamField>

<ParamField body="code" type="string">
  A 6-digit OTP code sent to the account's email address. Send this as the
  second request of the email-OTP flow (Path B step 2).

  <Expandable title="format">
    - Exactly 6 decimal digits (e.g. `"847213"`)
    - Alphanumeric or shorter values are rejected with `422 validation_error`
  </Expandable>
</ParamField>

<ParamField body="telegram_auth" type="object">
  A Telegram Login Widget payload. Send this as the second request of the
  Telegram re-auth flow (Path C step 2).

  <Expandable title="fields">
    <ParamField body="id" type="integer" required>
      Telegram user ID.
    </ParamField>
    <ParamField body="auth_date" type="integer" required>
      UNIX timestamp when the widget was authorised. Must be within the last 24 hours.
    </ParamField>
    <ParamField body="hash" type="string" required>
      HMAC-SHA256 signature produced by Telegram.
    </ParamField>
    <ParamField body="first_name" type="string">
      User's first name as returned by the widget.
    </ParamField>
    <ParamField body="username" type="string">
      User's Telegram username as returned by the widget.
    </ParamField>
    <ParamField body="photo_url" type="string">
      Profile photo URL as returned by the widget.
    </ParamField>
  </Expandable>
</ParamField>

## Behaviour

Deletion is **soft** — the account is not erased immediately. It is scheduled
for permanent removal 30 days after this request. During that window the user
can still sign in and cancel via
[`DELETE /profile/me/deletion`](/api-reference/profile/deletion/cancel-deletion).

The confirmation flow required depends on how the account was created. Call
[`GET /profile/me/deletion`](/api-reference/profile/deletion/deletion-method) first
to discover the path without spending a rate-limit token.

| `deletion_method` | Who it applies to | Flow |
|---|---|---|
| `password` | Accounts with a password set | Send `password` field → `200` |
| `email_otp` | No password, email linked | Empty body → `202` + OTP email → send `code` → `200` |
| `telegram` | Telegram-only (no password, no email) | Empty body → `202` → Telegram widget → send `telegram_auth` → `200` |

For a full walkthrough of each path see the
[Account Deletion Flow guide](/guides/profile/delete-account-flow).

Calling this endpoint when deletion is already scheduled returns `409 already_pending_deletion`.

## Rate Limiting

**10 requests per 1 hour per user**.

Clients that exceed this limit will receive a `429 Too Many Requests` response
with a `Retry-After` header indicating when they may retry.

## Responses

<AccordionGroup>
  <Accordion title="200 — OK">
    The account has been soft-deleted and is now scheduled for permanent removal
    after the 30-day grace period. Returned by Path A, Path B step 2, and Path C
    step 2.

    ```json
    {
      "message": "account scheduled for deletion",
      "scheduled_deletion_at": "2026-04-08T20:47:36Z"
    }
    ```

    From this point on, [`GET /profile/me`](/api-reference/profile/account/me)
    will include a `scheduled_deletion_at` field until the deletion is cancelled
    or the grace period expires.
  </Accordion>

  <Accordion title="202 — Accepted">
    Returned by Path B step 1 and Path C step 1 when a second confirmation step
    is required before deletion can be finalised.

    **Path B** — OTP has been emailed to the account address. Proceed with the
    `code` field in a follow-up request. The `expires_in` field tells the client
    how many seconds remain before the code expires — use it to render a
    countdown timer so the user knows to act promptly.

    ```json
    {
      "message": "verification code sent to your email",
      "auth_method": "email_otp",
      "expires_in": 900
    }
    ```

    **Path C** — render the Telegram Login Widget and re-submit with the
    `telegram_auth` payload.

    ```json
    {
      "message": "authenticate via Telegram to confirm deletion",
      "auth_method": "telegram"
    }
    ```

    Use [`GET /profile/me/deletion`](/api-reference/profile/deletion/deletion-method) to
    pre-fetch this value without triggering a DELETE request.
  </Accordion>

  <Accordion title="400 — Bad Request">
    Two causes:

    **Empty body on a password account** — password-protected accounts must
    always supply the `password` field. An empty body cannot trigger an OTP
    flow for such accounts.

    ```json
    {
      "code": "validation_error",
      "message": "password is required to delete a password-protected account"
    }
    ```

    **Malformed JSON** — the body could not be parsed.

    ```json
    {
      "code": "bad_request",
      "message": "malformed JSON body"
    }
    ```
  </Accordion>

  <Accordion title="401 — Unauthorized">
    Four causes:

    **Missing or invalid JWT:**

    ```json
    {
      "code": "unauthorized",
      "message": "missing user id in context"
    }
    ```

    **Wrong password (Path A):**

    ```json
    {
      "code": "invalid_credentials",
      "message": "invalid credentials"
    }
    ```

    **Telegram HMAC verification failed or `auth_date` older than 24 hours
    (Path C):**

    ```json
    {
      "code": "invalid_telegram_auth",
      "message": "telegram authentication is invalid or expired"
    }
    ```

    **Telegram payload `id` does not match the linked identity (Path C):**

    ```json
    {
      "code": "telegram_identity_mismatch",
      "message": "telegram identity does not match linked account"
    }
    ```
  </Accordion>

  <Accordion title="409 — Conflict">
    Deletion is already pending for this account. Cancel it first via
    [`DELETE /profile/me/deletion`](/api-reference/profile/deletion/cancel-deletion)
    before rescheduling.

    ```json
    {
      "code": "already_pending_deletion",
      "message": "account is already scheduled for deletion"
    }
    ```
  </Accordion>

  <Accordion title="413 — Request Too Large">
    The request body exceeds 1 MiB.

    ```json
    {
      "code": "request_too_large",
      "message": "request body too large"
    }
    ```
  </Accordion>

  <Accordion title="422 — Validation Error">
    Applies to Path B step 2 only.

    **Invalid code format** — `code` is not exactly 6 decimal digits:

    ```json
    {
      "code": "validation_error",
      "message": "code must be exactly 6 digits"
    }
    ```

    **No active token** — no pending account-deletion OTP exists. Either the
    code was never requested or it has expired. Restart from Path B step 1.

    ```json
    {
      "code": "token_not_found",
      "message": "verification token not found or expired"
    }
    ```

    **Incorrect code** — the submitted code does not match the stored token:

    ```json
    {
      "code": "invalid_code",
      "message": "invalid verification code"
    }
    ```

    **Token already consumed** — the OTP was already used:

    ```json
    {
      "code": "token_already_used",
      "message": "token has already been used"
    }
    ```
  </Accordion>

  <Accordion title="429 — Rate Limited">
    Three separate limits can produce this response.

    **User rate limit** — exceeded 10 requests per hour. Check the `Retry-After`
    header for when to retry.

    ```json
    {
      "code": "too_many_requests",
      "message": "too many requests — please slow down"
    }
    ```

    **Too many OTP attempts (Path B)** — the OTP token is locked after too many
    incorrect submissions. A new OTP can only be requested after the current
    token expires.

    ```json
    {
      "code": "too_many_attempts",
      "message": "too many attempts — token locked"
    }
    ```

    **Deletion token cooldown (Path B step 1)** — a deletion OTP has already been
    issued for this account and has not yet expired. Wait for it to expire or
    submit the code already sent before requesting a new one.

    ```json
    {
      "code": "deletion_token_cooldown",
      "message": "a deletion code has already been sent; please wait before requesting another"
    }
    ```
  </Accordion>

  <Accordion title="500 — Internal Server Error">
    An unexpected server-side error occurred.

    ```json
    {
      "code": "internal_error",
      "message": "internal server error"
    }
    ```
  </Accordion>
</AccordionGroup>
