After too many consecutive failed login attempts the account is time-locked. During the lockout window the correct password returns `429 login_locked` rather than a session. The unlock flow lets the account owner clear the lockout immediately without waiting for the window to expire.

---

## Overview

```
POST /auth/unlock  →  OTP sent to email (always 202)
PUT  /auth/unlock  →  email + code  →  200 (account unlocked)
POST /auth/login   →  sign in normally
```

Both endpoints share the same rate limit, so the combined total of requests to either endpoint from the same IP counts against one limit.

---

## What triggers a lockout

Ten consecutive wrong-password submissions on [`POST /auth/login`](/api-reference/auth/session/login) set a 15-minute lockout on the account. During that window, even the correct password returns:

```json
{
  "code": "login_locked",
  "message": "too many failed login attempts — please wait before trying again"
}
```

The `Retry-After` header tells the client exactly how many seconds remain. The user can wait it out or use the unlock flow to clear it immediately.

<Note>
  This flow covers **time-based lockouts** caused by failed login attempts. Accounts locked by an administrator require a separate admin action and cannot be self-served through this flow.
</Note>

## Full sequence

<Steps>
  <Step title="POST /auth/unlock — Request an unlock code">
    Send the locked account's email address. The server sends a 6-digit OTP to that address if the account exists and is currently time-locked.

    ```json Request
    {
      "email": "alice@example.com"
    }
    ```

    ```json Response — 202
    {
      "message": "if that email is registered and locked, an unlock code has been sent"
    }
    ```

    The response is always `202` regardless of whether the email is registered, the account is locked, or any other condition. Display the message as-is — never infer account state from the response.

    <Note>
      A **cooldown** suppresses duplicate issuances. If a code was already sent recently, no new one is dispatched but the response is still `202`. Tell the user to check their inbox before requesting again.
    </Note>
  </Step>

  <Step title="PUT /auth/unlock — Submit the OTP">
    The user enters the 6-digit code from their inbox. Submit it together with the email address.

    ```json Request
    {
      "email": "alice@example.com",
      "code": "837291"
    }
    ```

    ```json Response — 200
    {
      "message": "account unlocked successfully"
    }
    ```

    The lockout is cleared immediately. The user can log in right away.
  </Step>

  <Step title="POST /auth/login — Sign in">
    The account is now fully accessible. The user logs in with their existing credentials.

    ```json Request
    {
      "identifier": "alice@example.com",
      "password": "P@ssw0rd1!"
    }
    ```
  </Step>
</Steps>

## Error handling

| Screen | Response | Cause | Action |
|---|---|---|---|
| 2 | `422 validation_error` | Wrong code, code expired, already used, or no active token | Show "incorrect code", offer resend |
| 2 | `423 account_locked` | Admin lock set after token was issued | Direct to support — self-service unlock cannot clear an admin lock |
| 2 | `429 too_many_requests` | OTP attempt ceiling reached — code locked | Show "resend" button to issue a fresh code |
| 2 | `429 rate_limited` | Shared IP limit exceeded (3 req / 10 min across both endpoints) | Show `Retry-After` countdown |

---

## Shared rate limit

[`POST /auth/unlock`](/api-reference/auth/unlock/request-unlock) and [`PUT /auth/unlock`](/api-reference/auth/unlock/confirm-unlock) share the same rate-limit bucket. Three total requests from the same IP within 10 minutes — across either endpoint — exhausts the allowance. A fourth request to either endpoint returns `429`.

This means a user who sends two request-unlock calls and one confirm-unlock call has used their full allowance, and must wait for the window to reset before trying again.
