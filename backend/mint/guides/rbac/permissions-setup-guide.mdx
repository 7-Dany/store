This guide walks you through setting up roles and permissions for your store. Follow the steps in order — each step builds on the last.

---

## Overview

Access in the store is controlled by **roles**. A role is a named collection of permissions. You assign a role to a user, and that user inherits everything the role allows.

There are two kinds of roles:

- **System roles** — created automatically when the store is set up. They cannot be modified or deleted.
- **Custom roles** — created by you to match your team's structure.

---

## Step 1 — Create the role

Start by creating a role that represents a job or responsibility in your store.

Use [`POST /admin/rbac/roles`](/api-reference/rbac/roles/create-role) and give it a clear, descriptive name. Be specific — a name like `"Order Support Agent"` is better than `"Support"` because it makes the role's scope obvious to other admins.

```json
{
  "name": "Order Support Agent",
  "description": "Can view and manage customer orders on behalf of support."
}
```

Save the `id` from the response — you'll need it in the next steps.

---

## Step 2 — Find the permissions you need

Before assigning permissions, you need the `permission_id` for each one you want to grant. Use [`GET /admin/rbac/permissions`](/api-reference/rbac/permissions/list-permissions) to browse available permissions and note down the IDs.

Each permission has a `canonical_name` (e.g. `rbac:read`, `user:manage`) that describes exactly what it controls. Use these names to identify the right permissions for your role.

---

## Understanding permission capabilities

Every permission returned by [`GET /admin/rbac/permissions`](/api-reference/rbac/permissions/list-permissions) includes a `capabilities` object. Read this before building a grant — it tells you exactly which `access_type` and `scope` values are valid for that permission.

```json
// Example: product:manage
{
  "canonical_name": "product:manage",
  "capabilities": {
    "scope_policy": "any",
    "access_types": ["direct", "conditional", "request", "denied"]
  }
}

// Example: rbac:read
{
  "canonical_name": "rbac:read",
  "capabilities": {
    "scope_policy": "none",
    "access_types": ["direct", "denied"]
  }
}
```

### `capabilities.access_types`

Only the values listed here are accepted by [`POST /admin/rbac/roles/{id}/permissions`](/api-reference/rbac/roles/add-role-permission) for this permission. Passing any other value returns `422 access_type_not_allowed`. `"direct"` and `"denied"` are always present. `"conditional"` and `"request"` appear only when explicitly enabled for that permission.

### `capabilities.scope_policy`

Controls which `scope` value you may pass — and how the admin UI should render the scope field:

| `scope_policy` | UI guidance | Behaviour |
|---|---|---|
| `"none"` | Hide the scope field | Scope is not meaningful for this permission. The service silently normalises the stored value to `"all"` regardless of what you send. |
| `"own"` | Pre-fill `"own"`, render read-only | Only `scope: "own"` is accepted. Any other value returns `422 scope_not_allowed`. |
| `"all"` | Pre-fill `"all"`, render read-only | Only `scope: "all"` is accepted. Any other value returns `422 scope_not_allowed`. |
| `"any"` | Show selector with both options | Either `"own"` or `"all"` is accepted. Choose based on how much access the role should have. |

## Step 3 — Assign permissions to the role

Use [`POST /admin/rbac/roles/{id}/permissions`](/api-reference/rbac/roles/add-role-permission) for each permission you want to add.

Every grant requires you to choose an **access type** and a **scope**.

### Choosing an access type

| Access Type | When to use it |
|---|---|
| `direct` | The user always has this permission, no conditions apply. Use this for most cases. |
| `conditional` | The permission only applies when certain conditions are met. Use this for nuanced rules (e.g. only during business hours). Requires a `conditions` object. |
| `request` | The user must submit an approval request before the action executes. When a user hits an endpoint guarded by a `request`-type permission, the API returns `202` with `{ "code": "approval_required", "request_id": "..." }` instead of executing the action. Once an approver approves the request, the action proceeds. Use this for sensitive operations that need a second layer of approval. |
| `denied` | Explicitly blocks this permission, even if another role the user holds would allow it. Use this when you need a hard override. |

For most roles, start with `direct`. Only use `conditional`, `request`, or `denied` when you have a specific reason.

### Choosing a scope

| Scope | What it means |
|---|---|
| `own` | The user can only act on resources they created or own. |
| `all` | The user can act on any resource of this type, regardless of who owns it. |

A support agent who needs to view any customer's orders should get `scope: "all"`. A seller who should only manage their own listings should get `scope: "own"`.

### Writing a clear granted reason

The `granted_reason` field is required and is used for auditing. Write a sentence that explains *why* this permission is being granted — not just what it does.

Good: `"Support agents need to view all orders to assist customers."`  
Poor: `"orders read access"`

### Example grants

A simple `direct` grant for a permission with `scope_policy: "none"` (no scope field needed in the UI; the service normalises it):

```json
{
  "permission_id": "<rbac:read uuid>",
  "access_type": "direct",
  "scope": "all",
  "granted_reason": "Admins need to list roles and permissions."
}
```

A `request`-type grant for `job_queue:configure`, which has `allow_request = true` and `scope_policy: "none"`:

```json
{
  "permission_id": "<job_queue:configure uuid>",
  "access_type": "request",
  "scope": "all",
  "granted_reason": "Admin must request approval to pause or reconfigure job queues."
}
```

A `conditional` grant for `product:manage`, which has `scope_policy: "any"` and `allow_conditional = true`:

```json
{
  "permission_id": "<product:manage uuid>",
  "access_type": "conditional",
  "scope": "own",
  "conditions": { "region": "EU" },
  "granted_reason": "EU vendor may manage their own products within EU region constraints."
}
```

Repeat this for every permission the role needs.

If a permission is already granted to the role, the API returns `409 grant_already_exists`.
To change the `access_type` or `scope` of an existing grant, remove it first with
[`DELETE /admin/rbac/roles/{id}/permissions/{perm_id}`](/api-reference/rbac/roles/remove-role-permission),
then re-add it with the corrected values.

## Step 4 — Review the role's permissions

Once you've finished adding permissions, use [`GET /admin/rbac/roles/{id}/permissions`](/api-reference/rbac/roles/list-role-permissions) to review exactly what the role can do before assigning it to any users.

Check that:
- Every permission you intended to add is present
- The `access_type` and `scope` on each grant are correct
- No unnecessary permissions were added

---

## Step 5 — Assign the role to users

With the role ready, use [`PUT /admin/rbac/users/{user_id}/role`](/api-reference/rbac/user-roles/assign-role) to assign it to the appropriate users.

Each user holds **exactly one role** at a time. Assigning a new role replaces the previous one in a single operation — there is no intermediate state. The user immediately inherits all the permissions of their new role.

### Permanent vs temporary assignments

Omit `expires_at` for a permanent grant. The user retains the role until it is explicitly changed or removed.

```json
{
  "role_id": "<role uuid>",
  "granted_reason": "Permanent assignment for full-time order support staff."
}
```

Supply `expires_at` to create a temporary grant. The role stops being active the moment the timestamp passes — the user is treated as having no role until a new assignment is made.

```json
{
  "role_id": "<role uuid>",
  "granted_reason": "Covering for primary manager during leave — expires end of month.",
  "expires_at": "2024-07-31T23:59:59Z"
}
```

Temporary grants must expire between **5 minutes** and **90 days** from the time of the request. Values outside that range are rejected with `422`.

### Restrictions to be aware of

- **You cannot modify your own role.** Attempting to assign or remove a role for your own user ID returns `409 cannot_modify_own_role`.
- **The owner role cannot be assigned through this endpoint.** If the target user already holds the owner role, the request is rejected with `409 cannot_reassign_owner`. The owner role is only assignable via the bootstrap endpoint.

### Checking a user's current role

Use [`GET /admin/rbac/users/{user_id}/role`](/api-reference/rbac/user-roles/get-user-role) to confirm what role a user currently holds, when it was granted, and whether it is temporary.

```json
{
  "user_id": "018e4c3a-1b2d-7f4e-9c0a-2d3e4f5a6b7c",
  "role_id": "029f5d4b-2c3e-8a5f-0d1b-3e4f5a6b7c8d",
  "role_name": "Order Support Agent",
  "is_owner_role": false,
  "granted_reason": "Permanent assignment for full-time order support staff.",
  "granted_at": "2024-06-01T10:00:00Z",
  "expires_at": null
}
```

A `404 user_role_not_found` response means the user has no active role — either they were never assigned one, or a temporary assignment has expired.

### Removing a role

Use [`DELETE /admin/rbac/users/{user_id}/role`](/api-reference/rbac/user-roles/remove-role) to remove a user's current role assignment. After removal the user has no role and will be denied access to any permission-guarded endpoint.

The same restrictions apply: you cannot remove your own role, and you cannot remove the owner role through this endpoint. Additionally, the last active owner in the system cannot be removed — at least one owner must always exist.

## Step 6 — Grant direct permissions to a user

Sometimes a user needs access to a single permission that isn't covered by their role — but changing the role for every edge case isn't practical. Direct permission grants let you attach a specific permission directly to a user, independently of their role, for a fixed window of time.

Direct grants are **always temporary**. Unlike role assignments, `expires_at` is required on every grant. Once the grant expires the user loses the permission — there is no grace period.

### When to use direct grants

Use direct grants for narrow, time-limited exceptions — not as a substitute for good role design. Good candidates:

- A support agent who needs `user:manage` access for a single escalation this week, but whose role shouldn't permanently include it.
- A vendor who temporarily needs `product:manage` with `scope: "all"` to help migrate another seller's catalogue.
- Any situation where a one-off permission makes more sense than creating a whole new role.

If you find yourself granting the same permission directly to many users, that's a signal to add it to their role instead.

### Checking what a user already holds

Before granting, use [`GET /admin/rbac/users/{user_id}/permissions`](/api-reference/rbac/user-permissions/list-user-permissions) to see every active direct grant the user currently holds. Each grant shows its `id`, the permission's `canonical_name`, `scope`, `conditions`, expiry, and the reason it was granted.

```json
{
  "permissions": [
    {
      "id": "03a1f2b3-4c5d-6e7f-8a9b-0c1d2e3f4a5b",
      "canonical_name": "product:manage",
      "name": "Manage Products",
      "resource_type": "product",
      "scope": "own",
      "conditions": {},
      "expires_at": "2024-09-01T00:00:00Z",
      "granted_at": "2024-06-01T10:00:00Z",
      "granted_reason": "Temporary access to manage own product listings."
    }
  ]
}
```

### Granting a direct permission

Use [`POST /admin/rbac/users/{user_id}/permissions`](/api-reference/rbac/user-permissions/grant-user-permission). You must supply the `permission_id`, a `granted_reason`, and an `expires_at`. `scope` defaults to `"own"` if omitted.

```json
{
  "permission_id": "<product:manage uuid>",
  "granted_reason": "Temporary access to assist with catalogue migration — expires end of quarter.",
  "expires_at": "2024-09-30T23:59:59Z",
  "scope": "all"
}
```

The expiry window is enforced server-side: minimum **5 minutes**, maximum **90 days** from the time of the request. Values outside that range are rejected with `422`.

If the user already holds an active grant for the same permission, the API returns `409 permission_already_granted`. Revoke the existing grant first before re-granting with new parameters.

### Privilege escalation prevention

You can only grant permissions you hold yourself. If you attempt to grant a permission that is not in your own effective permission set, the API returns `403 privilege_escalation`. This applies regardless of your role — having `rbac:grant_user_permission` lets you perform grants, but not grants that exceed your own access.

### Scope and conditions

The same `scope_policy` rules from [Step 2](#understanding-permission-capabilities) apply to direct grants. Check the permission's `scope_policy` before deciding which scope to pass — an invalid scope returns `422 scope_not_allowed`.

You may optionally pass a `conditions` object to attach constraints to the grant, following the same structure used for conditional role permissions.

<Warning>
  **Conditional grants in V1 — content is not yet validated.** The `conditions` object is stored and passed through, but its contents are not validated against a schema at grant time. Any well-formed JSON object is accepted. Full condition schema enforcement is planned for a future release.
</Warning>

### Revoking a direct grant

Use [`DELETE /admin/rbac/users/{user_id}/permissions/{grant_id}`](/api-reference/rbac/user-permissions/revoke-user-permission) to remove a grant before its natural expiry. The `grant_id` comes from the `id` field in the list response. Revocation is immediate — the user loses the permission the moment the request succeeds.

Revocation is safe against client disconnects: the operation runs on a detached context, so a dropped connection mid-request cannot leave the grant in an ambiguous state.

## Keeping permissions clean

A few practices that keep your permission setup manageable over time:

**Design roles around job functions, not individuals.** Because each user holds exactly one role at a time, roles should represent a complete set of responsibilities for a given function. If a user needs capabilities from two roles, consolidate those permissions into a single role that covers both.

**Use `denied` sparingly.** An explicit deny overrides all grants, which can be surprising. Only use it when you need a guaranteed hard block, and document the reason clearly in `granted_reason`.

**Review roles periodically.** As your team changes, some roles may accumulate permissions that are no longer needed. Use [`GET /admin/rbac/roles/{id}/permissions`](/api-reference/rbac/roles/list-role-permissions) to audit what each role can do and use [`DELETE /admin/rbac/roles/{id}/permissions/{perm_id}`](/api-reference/rbac/roles/remove-role-permission) to remove anything that no longer applies.

**Don't modify system roles.** System roles are protected for a reason — they underpin core store behaviour. Build on top of them with custom roles instead.

**Expired temporary roles leave users with no access.** There is no grace period — the moment a temporary assignment expires, the user is locked out. If you use temporary grants, have a process in place to renew or replace them before they lapse.
